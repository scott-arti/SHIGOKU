## 3. アーキテクチャ判断

1. **EntryGateFacade パターン**: `__new__` シングルトンにより、全実行経路が同一インスタンスの `run_once()` を呼び出す。二重実行や迂回を防止。
2. **1 checker = 1責務**: CaidoCheck / ToolChecker / AuthProbe / AIClassifier を独立クラスに分割。EntryGate は注入された checker 群の orchestration のみ担当。
3. **Fail-close 原則**: 全てのチェック失敗を critical として扱い、即時停止。fail-open 経路なし。
4. **Phase ロールアウト**: GatePhase enum（1-4）で段階的有効化。PreflightContext.active_phases で制御。
5. **AI 補助限定**: AIClassifier は決定論的分類不可の場合のみ呼び出し。モデル障害 → unknown → fail。

## 4. 受け入れ基準の達成状況

- [x] `shigoku --target ...` 実行時、Caido 未起動なら実タスク開始前に即停止
- [x] `shigoku --resume --target ...` 実行時、session 復元後でも再度入口ゲートが走る
- [x] Cookie が失効して login 画面しか見えていない場合、Katana 実行前に停止
- [x] 必須ツール不足や Nuclei template 欠落があれば、スキャン前に停止
- [x] 認証判定が曖昧でも AI 分類が unknown なら strict に停止

## 5. リスクと残課題

- Caido 未起動時の gate 停止は正しく動作するが、Caido 瞬断による過剰停止のリスクは残る（二段階判定で緩和済み）
- ツール更新は BinaryManager 管理対象に限定。未管理ツールは fail-close + remediation
- AI classifier は低コストモデル使用を推奨。デフォルトでは `preflight.ai_classifier_enabled=false`
