# SGK-2026-0329 作業完了報告書

## 1. 要約
- Bug Bounty モードを injection 系 specialist（SQLi / XSS / CmdSSRF）の fail-closed デフォルトに変更した。
- HTTP method guard と payload guard を統合する共有セーフガードサービス `ExecutionSafeguardService` を新設し、`SmartRequest` の単一エントリポイントから呼び出す設計に統一した。
- 破壊的 SQL payload（DELETE FROM / UPDATE ... SET / INSERT INTO / DROP TABLE / TRUNCATE / ALTER）を Bug Bounty モードで共通遮断する `PayloadRiskPolicy` を実装した。
- CTF モードは既存の探索自由度を維持し、Bug Bounty との安全境界をコード・テストの両面で明確に分離した。
- 52 件の targeted unit test を追加し、既存 388 件の回帰テストに破損なし。

## 2. 変更内容

### 新規追加
- `src/core/security/execution_safeguard.py`
  - `SafeguardDecision` DTO（allowed / requires_hitl / reason_code / matched_rules / message）
  - `PayloadRiskPolicy`（破壊的 SQL パターン検出 + time-based ガバナンスフック）
  - `MethodRiskPolicy`（HTTP method ベースの allow/require_hitl 判定）
  - `ExecutionSafeguardService`（facade、fail-closed）
  - `get_execution_safeguard()` / `reset_execution_safeguard()` シングルトンファクトリ
- `tests/core/security/test_execution_safeguard.py`（52 tests）

### 修正
- `src/core/security/request_guard.py` — ドキュメント文字列を更新し、HTTP/HITL adapter としての責務を明示
- `src/core/security/__init__.py` — `SafeguardDecision`, `PayloadRiskPolicy`, `MethodRiskPolicy`, `ExecutionSafeguardService`, `get_execution_safeguard`, `reset_execution_safeguard`, `RequestGuard`, `get_request_guard`, `reset_request_guard` をエクスポート
- `src/core/infra/smart_request.py` — `execution_safeguard` パラメータを追加し、優先パスとして safeguard 評価を実行。既存 `request_guard` フォールバックを維持
- `src/core/agents/swarm/injection/smart_sqli.py` — デフォルト mode `ctf` → `bugbounty`、`get_execution_safeguard()` 使用
- `src/core/agents/swarm/injection/smart_xss.py` — 同上
- `src/core/agents/swarm/injection/smart_cmd_ssrf.py` — 同上

## 3. 設計判断
- `RequestGuard` は削除せず、HTTP/HITL endpoint-approval adapter として残した。`ExecutionSafeguardService` が facade として `MethodRiskPolicy` + `PayloadRiskPolicy` + `RequestGuard` を統合する。
- 破壊的 SQL パターンは `enhanced_ethics_guard.py` の既存定義を参照しつつ、`PayloadRiskPolicy` 内に専用の正本ルールを定義した。ルール定義の分散は将来的に一本化する。
- time-based SQLi（SLEEP / pg_sleep / WAITFOR DELAY 等）は `PayloadRiskPolicy.is_time_based_payload()` で検出可能だが、デフォルトでは遮断しない。中央管理用の `set_time_based_block()` フックを用意した。
- `interactive_bridge.py` は引き続き `get_request_guard()` を直接呼び出すが、同一シングルトンを `ExecutionSafeguardService` が参照するため、動作は一貫する。
