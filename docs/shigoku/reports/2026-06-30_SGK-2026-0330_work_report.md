## 2. 実装詳細

### 2.1 `smart_lfi.py` への safeguard 注入

**変更前:**
```python
mode = "ctf"  # CTF モードで POST リクエストを許可
...
self.smart_client = SmartRequest(network_client=self.network_client)
```

**変更後:**
```python
mode = "bugbounty"  # Bug Bounty fail-closed default
...
from src.core.security.execution_safeguard import get_execution_safeguard
safeguard = get_execution_safeguard(mode=mode)
self.smart_client = SmartRequest(network_client=self.network_client, execution_safeguard=safeguard)
```

`smart_sqli.py` / `smart_cmd_ssrf.py` / `smart_xss.py` と同一のパターンに統一。

### 2.2 `interactive_bridge.py` の HITL 初期化統一

**変更前:**
```python
from src.core.security.request_guard import get_request_guard
...
get_request_guard(mode=mode_val, hitl_callback=hitl_callback)
```

**変更後:**
```python
from src.core.security.execution_safeguard import get_execution_safeguard
...
get_execution_safeguard(mode=mode_val, hitl_callback=hitl_callback)
```

`get_execution_safeguard()` は内部で `get_request_guard()` を呼び出し、callback を引き回す。
これにより safeguard と request_guard が同一 singleton ツリーで管理され、二重初期化・別 cache namespace の発生を防止する。

### 2.3 テスト追加

4つの新規テストを追加:
- `test_smart_lfi_injects_execution_safeguard`: SmartRequest に ExecutionSafeguardService が注入されることの確認
- `test_smart_lfi_default_mode_bugbounty`: デフォルトモードが `bugbounty` であることの確認
- `test_smart_lfi_ctf_mode_override`: config で `mode=ctf` を指定した場合のオーバーライド確認
- `test_send_request_blocked_by_safeguard`: POST リクエストが callback 不在時に fail-closed ブロックされることの確認
- `test_send_request_allowed_by_safeguard`: GET リクエストが safeguard を通過することの確認

## 6. リスク

- **低リスク**: `smart_lfi.py` のデフォルトモードを `bugbounty` に変更したため、CTF 用途では明示的に `mode=ctf` を config で指定する必要がある。既存の CTF テスト/プロファイルでは config 経由で指定されていると想定。
- **低リスク**: `interactive_bridge.py` の `get_request_guard()` → `get_execution_safeguard()` 変更は、`get_execution_safeguard()` が内部で `get_request_guard()` を呼び出すため動作互換性を維持する。ただし、既存コードで `get_request_guard()` を独立して呼び出している箇所があれば、そちらは別途対応が必要（今回スコープ外）。
