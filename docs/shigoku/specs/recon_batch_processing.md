# 偵察バッチ処理の最適化仕様書 (Spec.md)

## 概要

SHIGOKUの偵察（Recon）プロセスにおいて、複数のターゲット（ドメインやURL）を個別にツール（Nuclei, httpx 等）でスキャンする際の起動オーバーヘッドを削減するため、ターゲットをリストとしてまとめ、一括実行（バッチ処理）する機能を導入します。

## 背景・目的

- **現状**: 1ターゲットごとにツールをプロセスとして起動している。ターゲット数が増えると起動時間が累積し、スキャン完了まで数時間かかる場合がある。
- **目的**: ツールがネイティブでサポートするリスト入力機能（`-l` オプション等）を活用し、プロセス起動回数を劇的に削減（1/N）する。

## 実装仕様

### 1. ツールラッパーのリスト対応

- **入力の拡張**: `BaseTool.run()` の `target` 引数が `str` だけでなく `List[str]` も受け入れ可能にする。
- **一時ファイル管理**: `List[str]` が渡された場合、`tempfile` を使用して一時的な `targets.txt` を生成し、ツールの `-l` オプションに渡す。
- **後方互換性**: 単一の `str` が渡された場合は、既存の `-u` (Nuclei) や直列実行を維持する。

### 2. セキュリティ・ガードレール (EthicsGuard) の適用

- **一括検証**: バッチファイルに書き込む直前に、リスト内のすべての要素に対して `EthicsGuard.check_action()` を実行する。
- **遮断時の挙動**: スコープ外のターゲットが含まれている場合、そのターゲットのみを除外してリストを作成する。すべてがスコープ外の場合は実行を中止する。

### 3. 対象ツール

- **Nuclei**: `-l` オプションを使用。一時ファイル経由での実行。
- **httpx**: 既存のファイルパス判定を強化し、`List[str]` からの一時ファイル生成に対応。

### 4. 結果のパース

- **JSONL ストリーミング**: バッチ実行の結果は `JSONL` 形式で取得し、1行ずつパースすることで、全体が終わる前でも部分的な結果を利用可能にする。

## 制約事項

- 一時ファイルは実行完了後（例外発生時も含む）に確実に削除すること。
- バッチサイズ（1回あたりのターゲット数）に上限（例: 50〜100）を設け、過度なリソース消費を防ぐ。
