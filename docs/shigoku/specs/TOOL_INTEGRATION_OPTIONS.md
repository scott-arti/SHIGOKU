# コマンドラインツールの統合方法案

SHIGOKU フレームワークに新しいコマンドラインツール（今回の `secretfinder` など）を統合し、「AI に使わせる」ための 3 つの実装パターンを提案します。

## 比較概要

| 方法                               | 実装難易度 | 信頼性・安全性 | 推奨シナリオ                                                 |
| :--------------------------------- | :--------- | :------------- | :----------------------------------------------------------- |
| **A. 専用ラッパークラス作成**      | 高         | 高 (推奨)      | 定常的・本格的に使用するメインツール、複雑な引数を持つツール |
| **B. Prompt Injection + LinuxCmd** | 低         | 低             | 一時的な実験、非常に単純なコマンド（例: `uptime`）           |
| **C. Recipe (YAML) 定義**          | 中         | 中             | 定型的な攻撃フロー、特定の状況下でのみ実行したい一連の手順   |

---

## A. 専用ラッパークラス作成 (推奨)

Python コードとしてツールクラスを実装し、入出力を厳密に管理する方法です。`ffuf.py` や今回作成した `SecretFinderTool` がこれに該当します。

### 実現手順

1. `src/tools/custom/` にツールクラスファイルを作成。
2. `src/core/factory.py` または `verify_secret_finder.py` のようにエージェントにロードさせる。

### メリット

- **構造化された引数:** LLM に対して「URL」と「モード」だけを指定させ、複雑なコマンドライン引数の組み立ては Python 側で安全に行えます。
- **出力解析:** ツールの標準出力をパースして、JSON などに構造化してから AI に返すことで、AI の理解度が格段に上がります。
- **安全性:** 不正な文字のサニタイズや、禁止された引数のフィルタリングが可能です。

### デメリット

- コーディングが必要です。

---

## B. Prompt Injection + LinuxCmd

既存の `LinuxCmd` ツールを使用し、プロンプトで「あなたは `secretfinder -u <URL>` というコマンドを使えます」と教える方法です。

### 実現手順

1. `src/core/factory.py` 内の `instructions` (プロンプト) に追記するか、`.agent/prompts/` のファイルを編集します。
   例: `"使用可能なツール: 'secretfinder <target>' コマンドが使用可能です。LinuxCmd経由で実行してください。"`

### メリット

- コードの変更が最小限で済みます。

### デメリット

- **Hallucination:** LLM が存在しないオプション（例: `--json` など）を勝手に捏造して実行し、エラーになる確率が高いです。
- **解析困難:** 出力がプレーンテキストのまま LLM に渡るため、トークン数を圧迫したり、重要な情報を見逃す可能性があります。

---

## C. Recipe (YAML) 定義

SHIGOKU の「Recipe 機能」を使い、特定のトリガー（例: 技術スタック検知）で実行される定型タスクとして定義します。

### 実現手順

1. `recipes/intel/secret_finder.yaml` を作成。

```yaml
name: "Secret Finder Scan"
trigger:
  tech_stack: "all"
steps:
  - tool: "linux_cmd"
    args:
      command: "secretfinder {{target}} -o /tmp/secrets.txt"
  - tool: "linux_cmd"
    args:
      command: "cat /tmp/secrets.txt"
```

### メリット

- **確実な実行:** 毎回必ず同じオプションで実行させたい場合に最適です（BugBounty のルーティンワークなど）。
- **No Code:** Python を書かずに YAML だけで追加できます。

### デメリット

- **自律性の欠如:** AI が「今はこれを試すべきだ」と判断して使うのではなく、条件合致時に機械的に実行されるだけになります。

---

## 結論

**`secretfinder` を「エージェントに自律的に使わせたい」のであれば、A 案（専用ラッパークラス）を推奨します。**

今回作成した `src/tools/custom/secret_finder.py` は A 案 に基づいています。これを少し調整して正式に組み込むのが最も高品質な統合となります。

## Deep Dive: ツール統合に関する高度な議論

ユーザーからの質問に基づき、より実践的で高度な設計指針を補足します。

### 1. ラッパーの定義：引数定義 vs 使用例 (Few-Shot)

**結論: 両方必要です。**

- **引数定義 (`to_schema`):** これは「文法」です。ツールが受け取れる型（文字列、数値、列挙型）を定義し、不正な形式の入力を防ぐ最低ラインです。
- **使用例 (Docstring/Prompt):** これは「教科書」です。LLM は定義だけでは「いつ、どのオプションを組み合わせるべきか」を迷います。
  - _Bad:_ 「`--scan-mode` というオプションがあります」
  - _Good:_ 「**通常は** `full` モードを使用してください。時間が限られている場合のみ `quick` を使います。例: `depth=2, mode='quick'`」

**ベストプラクティス:**
Docstring 内に「推奨される使用パターン（Golden Path）」を自然言語で記述することで、エージェントの成功率が劇的に向上します。

### 2. 専用ラッパーの隠れたメリット (Structure & Guardrails)

単にコマンドを実行するだけでなく、以下の価値を付加できます。

- **構造化 (Structuring):**
  `grep` や `secretfinder` の生のテキスト出力は、AI にとって「読みづらい」場合があります。ラッパー内で正規表現などを使い、結果を JSON (`{"found": true, "secrets": [...]}`) に変換して返すことで、AI の判断精度が向上します。
- **エラーハンドリングの抽象化:**
  ツール特有のエラーコード（例: exit code 139）をキャッチし、「メモリ不足です。スレッド数を減らして再実行してください」といった**AI が理解できる具体的なアドバイス**に変換して返すことができます。
- **ステート管理 (State Management):**
  認証トークンやセッションファイルの一時保存場所をラッパー内部で管理することで、AI にパス管理の手間をかけさせず、セキュアに保つことができます。

### 3. 高度なラッパー方式 (Advanced Patterns)

単なるコマンド実行以上の機能を持たせるパターンです。

- **Composite Tool (複合ツール):**
  1 回のツール呼び出しで「スキャン実行」→「結果のフィルタリング」→「フォーマット変換」までを一気通貫で行うツール。AI の思考ステップ（往復回数）を節約でき、トークンコスト削減と高速化につながります。
- **Micro-Agent Tool:**
  ツール内部でさらに小さな LLM 呼び出し（安価なモデル）を行い、「この出力結果は誤検知ではないか？」を自己判定してからメインエージェントに結果を返す「判断力付きツール」。

### 4. Linux コマンドの AllowList vs BlockList

**結論: セキュリティレベルを上げるなら AllowList (ホワイトリスト) に移行すべきです。**

- **現状 (BlockList):**
  「禁止されていないものは全て許可」。柔軟性は最高ですが、未知の危険なコマンドや、予期せぬオプションの組み合わせ（"Unknown Unknowns"）に対して脆弱です。
- **理想 (AllowList):**
  「許可されたコマンド以外は全て禁止」。
  例: `nmap` は許可するが、引数に `--script` (任意スクリプト実行) が含まれていたらブロックする、といった細かい制御が可能になります。
  **Shigoku のような攻撃ツールを扱うシステムでは、事故防止のため将来的には AllowList への移行を強く推奨します。**
