# 005. エンジン信頼性向上とコスト最適化

Date: 2026-01-05

## Status

Accepted

## Context

SHIGOKU のコアエンジンである `MasterConductor` および関連モジュールにおいて、以下の深刻な問題が特定された。

1.  **並列実行時のブロッキング**: `asyncio.get_event_loop()` の不適切な使用により、既存のイベントループが存在するスレッド（例: `asyncio.to_thread` 内）で `asyncio.run()` を呼び出すと `RuntimeError` が発生し、エージェントがクラッシュするか、デッドロックに似た状態に陥る。
2.  **リプランロジックの欠陥**: リトライ回数を管理するカウンターがグローバル変数であり、セッション全体で共有されていたため、特定のタスクが失敗すると他のタスクのリトライ権も消費してしまい、大規模なスキャンで回復機能が停止する。
3.  **タスクキューの枯渇と暴走**: 派生タスク（ReAct やリプランで生成されるタスク）が常にキューの先頭 (`insert(0, ...)`) に追加されるため、深さ優先探索のような挙動となり、派生タスクが無限に生成されると元のタスクが永遠に実行されない。
4.  **API コストの増大**: BizLogicHunter の Critic ループ（自己検証）がデフォルトで複数回実行され、また ReAct 観察機能がデフォルトで有効になっており、タスク成功ごとに LLM を呼び出すため、コストが想定以上に膨らむ。

## Decision

これらの問題に対処し、エンジンの安定性とコスト効率を向上させるため、以下の設計変更を採用した。

### 1. 安全な非同期実行ヘルパーの導入

並列実行時の安全性を確保するために、コンテキストアウェアな非同期実行ロジックを導入した。

- `asyncio.get_running_loop()` を使用して、現在のスレッドにアクティブなイベントループが存在するか確認する。
- ループが存在する場合: `concurrent.futures.ThreadPoolExecutor` を使用して、新しいスレッドで `asyncio.run()` を実行する。
- ループが存在しない場合: その場で `asyncio.run()` を実行する。

これにより、呼び出し元のコンテキスト（同期/非同期、メインスレッド/ワーカースレッド）に関わらず、非同期エージェントを安全に実行可能とした。

### 2. タスク個別リプランカウンターへの移行

グローバルなリトライ制限を廃止し、タスク個別の状態管理へ移行した。

- `Task` データクラスに `replan_depth: int` フィールドを追加。
- リプランによって生成された新しいタスクは、親タスクの `replan_depth + 1` を継承する。
- これにより、各タスクチェーンは独立して最大深度までリトライ可能となり、他のタスクの失敗に影響されなくなる。

### 3. タスクキュー管理の厳格化

無限ループと優先度逆転を防ぐため、タスク追加ロジックを一元化した。

- `_add_tasks` ヘルパーメソッドを導入し、すべてのタスク追加操作を集約。
- **優先度ソート**: タスク追加時に必ず `priority` 降順でソートし、常に最も重要なタスクが次に実行されるようにした。
- **派生タスク上限**: セッションごとの派生タスク総数に上限（デフォルト 20）を設け、無限ループ系バグに対する安全弁を設置した。

### 4. API コスト削減策

デフォルト設定を「安全側（低コスト）」倒し、LLM 利用を最適化した。

- **ReAct 観察**: `enable_react_observation` のデフォルト値を `False` に変更。明示的に有効化しない限り、タスク完了後の自動 LLM 呼び出しは行わない。
- **Critic ループ**: BizLogicHunter の自己検証ループを、修正前の「最大 3 回反復」から「1 回のみ実行、プロンプト最小化」に変更。プロンプトトークン数を約 90%削減し、実行回数も 66%削減した。

## Consequences

### Positive

- **安定性**: 並列モードでの実行が Python 3.10+ 環境でも安定し、`RuntimeError` によるクラッシュが解消された。
- **回復力**: 長時間実行される大規模スキャンにおいても、個々のタスクのリトライ機能が最後まで機能し続けるようになった。
- **安全性**: タスク生成の暴走によるメモリ枯渇や無限実行のリスクが、ハードリミットによって物理的に防止された。
- **経済性**: デフォルト設定での API 消費量が大幅に削減され、ユーザーが意図しない課金リスクが低減した。

### Negative

- **柔軟性の低下**: 派生タスクの上限により、正当な理由で大量のサブタスクを生成したいケース（例: 大規模なクローリング後の個別スキャン）で、スキャンが途中で打ち切られる可能性がある。（設定で上限緩和は可能）
- **精度とのトレードオフ**: Critic ループの回数削減により、LLM による自己修正の機会が減り、False Positive の除去率が若干低下する可能性がある。
