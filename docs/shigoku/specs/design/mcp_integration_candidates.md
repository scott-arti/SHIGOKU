# SHIGOKU MCP(Model Context Protocol) 導入検討ドキュメント

## 概要

本ドキュメントでは、SHIGOKUの各種内部ツール・コンポーネントをMCP（Model Context Protocol）サーバーとして外部化（分離）し、SHIGOKU本体のエージェントからMCPクライアントとして呼び出すアーキテクチャの導入検討内容をまとめる。

VAPTエンジンという特性上、MCPによる「サンドボックス化」や「疎結合な分散処理」は非常に強力なメリットをもたらすが、パフォーマンス面での代償もあるため、導入対象とその効果を精査した。

---

## 検討対象のMCPサーバー一覧と評価

SHIGOKU内部のPythonラッパーから切り離し、専用のMCPサーバー化することで特に効果が高いと期待されるコンポーネントを以下に挙げる。

### 1. プロキシツール連動MCP (Caido / Burp Suite)

- **現在の状態**: `caido_importer.py` などでエクスポートされた静的なJSONログを解析。
- **MCP化のメリット (特大)**:
  - **リアルタイムな動的解析**: エージェントがMCP経由で「現在プロキシを通っている特定ドメインの履歴を抽出して」「このリクエストのCookieを削って再送して」といった動的なAPI操作を直接行えるようになる。
  - 静的ファイル連携の手間が省け、エージェント主導の動的テストが可能になる。
- **MCP化のデメリット**:
  - 大量のHTTP履歴をJSONでやり取りするため、通信量のコントロールとコンテキストウィンドウの消費対策が必須。

### 2. 重スキャン・エンジンMCP (Nuclei, ffuf, SQLMap等)

- **現在の状態**: `subprocess` でローカル実行し、完了を待機する。環境(Docker/Local)ごとのパス解決(`_resolve_template_path`)などの複雑なロジックをPythonで抱えている。
- **MCP化のメリット (大)**:
  - **非同期・分散処理の実現**: 数時間かかるスキャンを外部のMCPノードに投げ、SHIGOKU本体のメインループやスレッドプールを解放できる。
  - **依存関係の隔離**: 実行系(Go、Javaなど)や設定パスの問題を、コンテナ化されたMCPサーバー内に閉じ込めることができる。
- **MCP化のデメリット**:
  - スキャンの進行状況（Stdoutのリアルタイムプログレス）の追跡や、細かな実行停止・再開などの複雑なプロセス制御が難しくなる。

### 3. サンドボックス実行環境MCP (ペイロード生成・JS解析)

- **現在の状態**: 一部コマンドをコンテナやローカルで実行しているが、難読化されたJSや危険なスクリプトを安全に解析するハードルが高い。
- **MCP化のメリット (特大・セキュリティの要)**:
  - **完全な隔離（Air-Gapped Sandbox）**: ネットワークから隔離されたDockerコンテナ上のMCPを作り、そこに疑わしいJSの実行や解析を丸投げできる。万が一ゼロデイ攻撃やコマンドインジェクションを受けMCPコンテナが乗っ取られても、SHIGOKUの本体プロセス（APIキーやデータベース）は完全に保護される。
- **MCP化のデメリット**:
  - 実行環境の立ち上げと破棄のライフサイクル管理や、コンテナのインフラ整備が別途必要になる。

### 4. 脅威グラフ(Threat Graph) MCP (Neo4j)

- **現在の状態**: `neo4j` Pythonドライバを用いて、スキーマスキーマやコネクション管理、Cypherクエリの発行を直接Python側でハードコードしている。
- **MCP化のメリット (中〜大)**:
  - ** Graph RAGへの進化 **: 複雑なCypherクエリの発行ロジックをMCP側で隠蔽し、自然言語での「このエンドポイントから管理画面へ繋がる攻撃パスを教えて」といった高度なGraph RAGインターフェースをエージェントに提供できる。
- **MCP化のデメリット**:
  - データのエクスポートやインポートなど、大量のバッチ処理を行う際には通信プロトコルのオーバーヘッドが大きくなる場合がある。

---

## 総合的なメリットとデメリット（総括）

### 🌟 メリット (Pros)

1.  **究極のサンドボックス化・安全性向上**: 最も危険な操作（ブラウザ操作、悪意あるレスポンスのパース、JS実行）を完全にコンポーネント分離できる。SHIGOKU本体プロセスが乗っ取られるリスクを激減。
2.  **インフラスケーラビリティの獲得**: 攻撃やスキャン処理のマイクロサービス化により、大量のターゲットに対する分散スキャンアーキテクチャへと進化できる。
3.  **環境依存性の排除 (Dependency Hellの解消)**: ツールのインストールパス、依存ライブラリのバージョン等の問題をMCPコンテナ内に閉じ込め、SHIGOKU本体をクリーンに保てる。

### ⚠️ デメリット (Cons)

1.  **通信オーバーヘッドとレイテンシの増加**: メモリ上での直接呼び出し（Pythonライブラリ）に比べ、JSON-RPC等によるプロセス間通信が発生するため、Playwrightでの細かいDOMトラバース等の高頻度な操作は大幅に遅延する可能性がある。
2.  **APIの制約と微細な制御の喪失**: VAPT特有の「通信の途中でパケットを書き換える」「TCPレベルでパケットを作る」といった低レベルな操作を、抽象化されたMCPのツール定義（Schema）に落とし込むのは非常に困難。
3.  **インフラ管理の複雑化**: デプロイ時に「SHIGOKU本体」「Neo4j MCP」「Nuclei MCP」「Sandbox MCP」など複数のサービスをオーケストレーションする必要が生じる。

---

## 今後の導入検討方針

「何もかもをMCPにする」のではなく、**「処理の特性」に応じたハイブリッドアーキテクチャ**が最適解である。

1.  **Playwright等、微細で高速な制御が必要なものはPythonラッパーで残す**のが現状では手堅い。
2.  導入の優先度としては、安全性が劇的に向上する**「Sandbox実行環境MCP」**と、動的テスト能力が根本から変わる**「Caido (Proxy) 連携MCP」**から検討・PoC（概念実証）を行う。
3.  将来のエンタープライズ化（並列分散スキャン）を見据え、Heavy Scanners (Nuclei / ffuf 等) のMCPワーカー化を段階的に進める。
