# IDOR (Insecure Direct Object Reference) / BOLA スキャナの実装課題と将来のロードマップ

現在の `IdorHunterSpecialist` の実装は基礎的な PoC レベルに留まっており、モダンな Web アプリケーションの複雑な認可モデルを完全にテストするには以下のパターンに対応していく必要があります。

## 1. 識別子（ID）の位置とフォーマットの網羅性不足

現在の実装は「URL パス内の数値」のみを対象としています。

- **課題**:
  - **リクエストボディ内の ID**: JSON (`{"user_id": 123}`), XML, GraphQL の Variables などの操作に対応していない。
  - **クエリパラメータ内の ID**: `/api/data?id=123` などの URL クエリパラメータ操作が漏れている。
  - **HTTP ヘッダー内の ID**: カスタムヘッダー（例: `X-User-Id: 123`）の操作ができない。
  - **複雑な識別子**: 数値以外の識別子、特に UUID/GUID、メールアドレス、ハッシュ化された文字列などへの対応が急務。

## 2. 真の認可（Authorization）テストの欠如（マルチコンテキストの不足）

現在の「ID 操作テスト」は単一の認証コンテキスト内で実行されています。

- **課題**:
  - **BOLA の正確な検知**: 真の権限昇格テストを行うには、「ユーザーAのトークンでユーザーBのリソースにアクセスする」クロスユーザーテスト（Matrix Testing）が不可欠。現在の仕組みでは、アクセスできた別 ID が「自分自身の別のリソース」か「他人のリソース」かを正確に判別できない。
  - **BFLA (Broken Function Level Authorization)**: ID だけでなく、機能レベルでの認可不備（例: 一般ユーザーによる `/api/admin/*` へのアクセス）のテストが不足している。

## 3. 検知ロジックの脆弱性（過検知と検知漏れ）

現在の検知は「200 OK」と「レスポンス長の類似性」に依存しています。

- **課題**:
  - **過検知 (False Positive)**: パブリックなリソースに対して 200 OK が返る場合や、カスタムエラーハンドラが常に 200 OK でエラーJSONを返す場合、誤検知が発生する。
  - **検知漏れ (False Negative)**: 認証なしテストで一部の機密データだけが欠落した JSON が返ってきた場合など、レスポンス長が大きく変わるが脆弱性として扱うべきケースを拾いきれない。

## 4. 破壊的操作（POST/PUT/DELETE）の危険性

現在の実装は `task` で指定された HTTP メソッドをそのまま用いて操作テストを実行します。

- **課題**:
  - ターゲットが `DELETE` メソッドの場合、自動的に推測した別 ID を送信してしまい、本当に他人のデータを意図せず削除してしまう危険性がある。
  - 状態変更を伴う操作に対しては、安全な方法（読み取り専用に変換する等）での事前確認が必要。

## 5. Blind IDOR / Second-order IDOR への未対応

- **課題**:
  - ID を操作したリクエストのレスポンスには結果が現れず、後続の別機能（例えば非同期のレポート生成やエクスポート機能）で初めてデータが漏洩するパターンの検知ができない。

## 今後のロードマップ / 完了状況

これらの課題を解決するため、以下の順序で Specialist を強化（あるいは新規 Agent を追加）していく想定です。

1.  ✅ **Parsing & Discovery の強化**: URL、Query、Body จากあらゆる推測可能な ID (UUID等) を抽出・操作する機能の追加。
    - _完了_: URL パス内の UUID 抽出、および JSON Body オブジェクト内の指定キー (`id`, `uuid` 含む) の再帰的抽出と操作を実装。
2.  **マルチ・セッション検証 (Matrix Testing) の導入**: `SharedWorkspace` に「User A」「User B」「Admin」などの複数コンテキストを保持し、クロスで検証を行うアーキテクチャの整備。
3.  ✅ **セマンティックベースの検知ロジック**: ステータスコード依存を脱却し、LLM や専用ツールによるレスポンス内容の知的評価（機密情報の漏洩有無判定など）の導入。
    - _完了_: Antigravity 改良版の `SecretFinder` を統合し、IDOR / Unauth アクセスの結果生じた 200 OK のレスポンスボディ内に Token, Secret, Private Key が露出していないかを JSON フォーマットで高精度にパース・検証するロジックを実装。
4.  ✅ **Safe Mode 定義**: 破壊的メソッドに対する安全なテスト手法の確立。
    - _完了_: デフォルトで `POST`/`PUT`/`DELETE`/`PATCH` などの破壊的メソッドを検知した場合はスキップするセーフモード機能 (`safe_mode=True`) を導入。
