# Spec: エンジン・パフォーマンス最適化

## 1. 概要

SHIGOKUエンジンの実行効率、安定性、およびスケーラビリティを向上させるため、並列処理アーキテクチャとリソース管理の抜本的な改善を行います。

## 2. 変更内容

### 2.1 イベントループとリソース管理の刷新

- **単一イベントループの維持**: アプリケーションライフサイクル全体で1つの `asyncio` ループを使用し、リソース（セッション等）の瞬きを防止します。
- **SwarmDispatcherのクリーンアップ**: シャットダウン時に全ての Swarm インスタンスを確実にクローズするよう、`MasterConductor.shutdown` を強化します。

### 2.2 TaskQueue の高度化

- **Redisへの対応準備**: ストレージエンジンを抽象化し、SQLiteとRedisを切り替え可能にします。
- **プリエンプティブ・スケジューリング**: 実行中のタスクよりも明らかに高い優先度を持つタスクが発生した場合、次の実行枠を奪取するロジックを実装します。

### 2.3 並列ディスパッチと動的スケーリング

- **ParallelOrchestratorの統合**: `MasterConductor` がタスクを順次実行するのではなく、カテゴリ別に並列実行できるようにします。
- **リソースベースの動的調整**: `SystemResourceManager` が CPU/メモリ/帯域を監視し、`ParallelOrchestrator` のワーカー数をリアルタイムで増減させます。

## 3. 挙動への影響

- 大規模スキャン時のメモリ消費量の低減。
- 致命的な発見（Adminパネル発見等）に対するレスポンス速度の向上。
- スループット（単位時間あたりのリクエスト数）の大幅な向上。

## 4. 制約と安全性

- **EthicsGuard 一致**: 全ての並列リクエストは `EthicsGuard` のスコープチェックを通過しなければなりません。
- **レート制限の遵守**: 並列実行時も `AdaptiveRateLimiter` によるターゲット単位の RPS 制限を厳格に守ります。
