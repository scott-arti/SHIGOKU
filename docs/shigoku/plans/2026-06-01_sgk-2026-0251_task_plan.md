# 実装計画書：脆弱性チェーン高精度化実装計画

## 1. 達成したいゴール（ユーザー視点）
- [ ] スキャン中に新規 Finding が追加された時、攻撃連鎖候補が自動で更新されること（取りこぼし防止）。
- [ ] 重要Finding追加時・件数閾値到達時・終了前に、連鎖候補が再評価され `draft -> confirmed -> actionable` に昇格/降格されること。
- [ ] レポート前最終ゲートで、再現証跡が不足する連鎖は提出対象から除外されること。
- [ ] Top-N（既定 5 件）に表示されるチェーンが「再現可能かつ報酬期待の高い順」になること。

## 2. 全体像とアーキテクチャ
- **対象コンポーネント/ファイル一覧:**
  - `src/core/intelligence/chain_builder.py`: （修正）グラフ探索・状態機械・証拠駆動スコアリングの中核。
  - `src/core/engine/master_conductor.py`: （修正）チェーン評価トリガー、最終ゲート、重複抑止、通知連携。
  - `data/attack_chain_rules.json`: （修正）攻撃目的別テンプレート、前提条件、期待インパクト/期待報酬重み。
  - `tests/core/intelligence/test_chain_builder.py`: （修正）ルール推論・状態遷移・スコア検証。
  - `tests/core/engine/test_master_conductor_intervention_gate.py`: （修正）アクション前ゲートの回帰テスト。
  - `tests/scripts/verify_chaining_flow.py`: （修正）E2Eでの連鎖生成・昇格・抑止動作確認。
- **データの流れ / 依存関係:**
  - Finding Stream (`handle_finding`) -> 軽量候補更新 (`draft`) -> バッチ再評価（グラフ探索） -> `confirmed` 作成 -> 最終ゲート検証 -> `actionable` を report/escalate。
  - 連鎖判定は既存 Finding モデルと tags/additional_info を再利用し、永続化と通知は既存フローを利用。

### 2.1 フェーズ構成と確認対象
- Phase 0（基盤整備 / Step 1〜6）:

  - 確認: `chain_event` 必須項目充足、旧新DSL互換、重複チェーン抑制。
- Phase 1（価値創出MVP / Step 7〜18）:
  - 対象: 主体モデル、スコアリング、反証、提出品質、ゴール状態化。
  - 確認: actionableの品質、再現性、提出可能性、影響説明の妥当性。
- Phase 2（運用安定化 / Step 19〜27）:
  - 対象: テスト拡張、E2E/負荷、監査ログ、停止条件、運用Runbook。
  - 確認: 監視可能性、障害時degrade、提出先フォーマット適合。
- Phase 2.5（AI仮説層 / Step 27A〜27D）:

- Phase 3（高度最適化 / Step 28〜37）:
  - 対象: POMDP/MCTS/確率前提/因果介入/校正ループ。
  - 確認: 成功率改善がコスト増を上回ること、校正誤差（ECE）が閾値以内であること。

## 3. 具体的な仕様と制約条件
- **入力情報 (Input):**
  - `findings: list[Finding]`
  - `chain_rules: list[AttackChainRule]`（signal, preconditions, objective, impact_weight, bounty_weight）
  - `runtime_context`（mode, scope, replay_evidence）
- **出力/結果 (Output):**
  - `AttackChain`（`state`, `confidence`, `impact_score`, `bounty_score`, `priority_score`, `component_findings`, `missing_evidence`）
  - 最終的に `actionable` のみ `Finding` として昇格し既存通知/保存へ流す。
  - 証跡不足・スコープ違反は `confirmed` から `draft` へ降格、または `blocked` として保留。
- **制約・ルール:**
  - 連鎖状態は `draft / confirmed / actionable / blocked` の4状態で管理。
  - 連鎖構築時に前提条件を必須評価:
    - `auth_level`（unauth/user/admin）
    - `user_interaction`（none/click/social）
    - `same_origin`（true/false）
    - `asset_scope`（in_scope only）
    - `primitive`（read/write/exec/pivot）
  - 探索は「状態遷移グラフ」で実施し、2点固定でなく 2〜5 ノードの経路を評価。
  - `confidence` は証拠駆動で計算:
    - 再現回数
    - HTTP差分の強度
    - 機密情報/権限昇格の実証有無
    - 独立観測（別エージェント/別手法）数
  - `priority_score = w1*impact + w2*bounty + w3*confidence - w4*assumption_penalty`。
  - EthicsGuard と scope 制約を最終ゲートで必ず再検証。
  - 同一 chain_key の再発行は禁止（既存 dedup を維持）。
  - 既存公開挙動を壊さない段階導入（feature flag でオン/オフ可能）。

### 3.1 攻撃目的カテゴリ（バリエーション拡張）
- ATO（アカウント乗っ取り）
- Privilege Escalation（権限昇格）
- Sensitive Data Exfiltration（機密取得）
- Financial/Business Abuse（クーポン・決済不正）
- Internal Pivot / Lateral Movement（SSRF起点横展開）

### 3.1.1 業種別テンプレート分岐（発見機会拡大）
- EC:
  - coupon abuse -> wallet manipulation -> order takeover
  - stored xss -> admin action csrf -> price/inventory tamper
- Fintech:
  - idor -> beneficiary swap -> unauthorized transfer
  - race condition -> double spend -> settlement inconsistency
- SaaS/B2B:
  - tenant idor -> cross-tenant data read -> role escalation
  - oauth redirect abuse -> session fixation -> org admin pivot
- `data/attack_chain_rules.json` に `industry` フィールドを追加し、対象プロジェクトのタグから優先テンプレを自動選択する。

### 3.2 昇格条件（状態機械）
- `draft -> confirmed`:
  - 前提条件整合 + 経路長2以上 + ルール一致 + 最低証跡スコア到達 + 時間軸制約成立。
- `confirmed -> actionable`:
  - 再現証跡（PoC or replay）あり + スコープ適合 + 倫理ゲート通過 + 反証チェック合格。
- `actionable -> blocked`:
  - スコープ逸脱、再現失敗継続、または前提崩壊。

### 3.3 精度向上6点の実装仕様
- Negative Chain学習:
  - `NegativeChainRecord`（`rule_id`, `component_fingerprint`, `failure_reason`, `observed_at`, `decay_ttl`）を保存。
  - near-miss/不成立パターンは `assumption_penalty` を増やし、同型チェーンの優先度を減点。
- 時間軸イベント制約:
  - `timeline_constraints`（`after_login`, `post_token_refresh`, `checkout_window` など）をルールとチェーン双方で扱う。
  - 条件未成立なら `draft` 維持、成立時のみ `confirmed` へ昇格。
- Asset Graph連携:
  - ノードに `asset`, `auth_boundary`, `tenant_boundary`, `admin_surface` を追加し、境界横断を経路評価に反映。
  - 2〜5ノード探索時に「境界越え価値」を impact/bounty に加点。
- Exploitability重視証拠:
  - `exploitability_evidence` として `state_change_success`, `cross_user_impact`, `persistence_success` を収集。
  - いずれかが成立したチェーンを優先表示。
- 業種別テンプレ適用:
  - EC/Fintech/SaaS の目的別連鎖をルールセットで分離し、targetタグで切り替え。
  - 不明時は共通テンプレ + 各業種上位1件をバックオフ適用。
- 反証チェックリスト:
  - `falsification_checks` を自動生成（例: 再ログイン後再現不可、権限境界誤判定、キャッシュ起因誤検知）。
  - `confirmed -> actionable` の必須ゲートにする。

### 3.4 追加4点の実装仕様
- Chain DSL化:
  - `data/attack_chain_rules.json` を段階移行し、`dsl_version` を導入。
  - DSL要素: `preconditions`, `transitions`, `required_evidence`, `falsification`, `industry`, `objective`。
  - 既存ルールは互換ローダで読み込み、新DSLへ順次移行する。
- Counterfactual Scoring:
  - 各候補チェーンで「構成要素を1つ除いた場合の成立可否」を評価。
  - 代替経路で成立する場合は寄与度を下げ、偶然一致を減点。
  - `counterfactual_penalty` を `priority_score` に反映する。
- Active Probing for Missing Links:
  - `draft/confirmed` で不足している証拠リンクを特定し、追加検証タスクを自動生成。
  - 例: CSRF実在確認、権限境界再試験、トークン再利用可否チェック。
  - 実行回数/対象数に上限を持たせ、過負荷を防止する。
- Program-specific Memory:
  - プログラム単位で `successful_chains`, `failed_chains`, `high_yield_patterns` を保存。
  - 次回スコアリング時に prior として利用し、対象依存の命中率を引き上げる。
  - 保存時は chain fingerprint を匿名化し、機密データを残さない。

### 3.4.1 AIハイブリッド連鎖推論（Phase 2.5 追加）
- 役割分離:
  - AIは `draft` 候補生成と `missing_evidence`/`recommended_probe`/`business_impact_hypothesis` の補助推論に限定する。
  - `confirmed/actionable/blocked` の最終状態判定、監査、提出可否、停止判定は既存コードが保持する。
- 新規コンポーネント:
  - `src/core/intelligence/chain_proposal.py` に `ChainProposalEngine` と `LLMChainProposalEngine` を追加する。
  - `src/core/intelligence/chain_builder.py` は `proposal_engine` を optional で受け取り、`analyze_hybrid()` から既存 `analyze()` と AI候補をマージする。
- AI出力契約:
  - 自由文ではなく strict JSON のみ受理する。
  - 必須項目: `objective`, `path`, `required_findings`, `missing_evidence`, `exploitability_evidence`, `foothold_reliability`, `expected_attempts_to_success`, `business_impact_hypothesis`, `recommended_probe`, `reasoning_summary`。
  - `reasoning_summary` は監査用の短い要約のみとし、長い思考ログは保存しない。
- 安全制約:
  - AI候補は常に `draft` から開始し、既存 `promote_chain()`/`validate_report_payload()`/`emit_chain_audit_record()` を通過しない限り昇格しない。
  - invalid JSON、timeout、予算超過、モデル障害時は `proposal skipped` とし、既存ヒューリスティックへ fail-safe fallback する。
  - `canonical_report_payload` は AIが直接生成せず、既存 builder が組み立てる。
- 呼び出し条件:
  - finding が2件以上、かつ既存 `analyze()` が空または低信頼の場合。
  - high severity finding を含む場合。
  - `pre_action_gate` 前の最終再評価時。

### 3.5 フェーズ別仕様確認ポイント
- Phase 0 で確認する仕様:
  - 入力データ契約（`findings`, `chain_rules`, `runtime_context`）の必須項目と欠損時挙動。
  - `draft/confirmed/actionable/blocked` の遷移前提と `chain_key` 同値性正規化。
- Phase 1 で確認する仕様:
  - 攻撃目的カテゴリ/業種テンプレの適用優先順位。
  - `confirmed -> actionable` の反証ゲート、`business_impact_sentence`、`goal_state_assertions` の必須化。
- Phase 2 で確認する仕様:
  - fail-open/fail-closed、監査ログ最小項目、提出先フォーマット適合。
  - 外部障害時 `defer`、スコープ逸脱時 `blocked` の遷移一貫性。
- Phase 3 で確認する仕様:
  - belief state と MCTS の段階導入条件。
  - 成功確率校正（ECE）と因果介入妥当性の継続補正。
- Phase 2.5 で確認する仕様:
  - `LLMChainProposalEngine` の strict JSON 契約、timeout/fallback、feature flag の効き方。
  - AI候補が `draft` からしか始まらず、既存 gate を迂回できないこと。
- Phase 1/2 共通で確認する仕様:
  - `canonical_report_payload` をレポート品質ゲートの正本スキーマとし、必須項目を `title`, `business_impact_sentence`, `reproduction_steps`, `boundary_cross_proof`, `victim_impact`, `remediation`, `falsification_result`, `goal_state_assertions` に固定する。
  - `Haddix formatter` と `platform_integration` は `canonical_report_payload` から派生生成し、提出先差分は adapter 層のみで吸収する。
  - 監査の正本は `AuditLogger`、意思決定根拠の正本は `DecisionTracer` とし、攻撃チェーン固有項目は既存2系統へマッピングして保存する。
  - KPI比較は `benchmark_manifest` を正本とし、corpus、seed、headers、session policy、label snapshot、比較期間を固定する。
- Program-specific Memory は「永続ストア + 実行時キャッシュ」の二層構成とし、永続ストアを正本、in-memory を高速化用キャッシュとする。
- Phase 2.5/3 共通で確認する仕様:
  - AI候補は `chain_key` canonicalization と既存 dedupe に統合される。
  - モデル差し替え時も `benchmark_manifest` と shadow comparison で比較可能である。

### 4.0 実行ルール（8点改善の反映）

- `shadow` では既存ヒューリスティックの出力を変更せず、比較メトリクスのみ記録する。
- rollback 条件は `成功率悪化`、`CPU time per confirmed chain 増加`、`ECE 悪化` のいずれか成立時とする。
