# 実装計画書：Active計画書の統合と並列実行バンドル整理

- [x] `docs/shigoku` の active な計画書を、独立実行が必要なものと execution bundle に吸収すべきものへ整理できること。

- **データの流れ / 依存関係:**
  - active plan inventory -> overlap / dependency review -> absorbed subtasks を `backlog` 化 -> execution bundle と parallel-safe group を台帳と計画書へ反映。

## 3. 具体的な仕様と制約条件
- **入力情報 (Input):**
  - `task_registry.yaml` 上の `status=active` かつ `doc_type in {plan, subtask_plan}` の計画書群。
  - 親子関係、`related_docs`、計画本文のスコープ記述。
- **出力/結果 (Output):**
  - 吸収対象の `backlog` 化。
  - primary 実行単位として扱う execution bundle の定義。
  - 並列実行してよい計画群と、その前提条件。
- **制約・ルール:**
  - scope が上位計画の後段フェーズまたは詳細設計に過ぎないものだけを吸収対象にする。
  - 吸収は `done` ではなく `backlog` とし、未実装である事実を隠さない。
  - 親計画の実行順を変える場合でも、既存 task ID と primary doc は保持する。
