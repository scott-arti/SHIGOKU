# Coding Rules

Use this file for implementation, refactoring, and code quality changes.

## Quality baseline

- Do not produce code that merely appears to work. Generated code must be explicit, tested, maintainable, and safe.
- Do not excuse low-quality code because it was generated by an AI agent.
- Follow existing project patterns before adding new abstractions.
- Do not introduce redundant custom logic when Python, the standard library, or an existing project helper already provides the behavior.
- Do not change public behavior without updating relevant tests and documentation.
- If a project quality scanner exists, run it when the task changes generated or agent-written code. Do not reference tools that are not present in the repository.

## Code quality principles

When implementing code:

1. Make the behavior correct first.
2. Make the intent obvious.
3. Remove duplication only when the duplicated concept is truly the same.
4. Prefer fewer moving parts over clever abstraction.
5. Keep changes local: a small requirement change should require a small code change.
6. Do not introduce a framework, pattern, or abstraction unless it removes current complexity.
7. After tests pass, review the diff and remove accidental complexity.

## Error handling

- Do not use bare `except:` or empty `except` blocks.
- Do not catch broad `Exception` unless the boundary requires it.
- Catch only expected exception types.
- Re-raise unexpected exceptions.
- Do not hide unknown failures behind broad fallback behavior.
- Add context when re-raising or returning errors, but do not expose secrets.

## File operations

- For file deletion, ignore only `FileNotFoundError` when a missing file is safe.
- Do not ignore `PermissionError`, unexpected `OSError`, or I/O failures.
- Prefer `pathlib.Path` for path manipulation unless existing code uses another convention.

## Process and subprocess handling

- Do not ignore subprocess failures.
- Use `check=True` when command failure should stop the operation.
- Capture and report stderr when useful, but redact secrets.
- For process termination, ignore only `ProcessLookupError` when the process is already gone.
- Do not ignore permission or unexpected process errors.

## Async and timeout handling

- Do not use arbitrary sleeps as synchronization.
- Use explicit timeouts for external I/O, subprocesses, network calls, and long-running operations.
- Handle `asyncio.TimeoutError` only where timeout recovery is intentional.
- Do not swallow exceptions in background tasks.

## Data and type safety

- Prefer precise types for public functions, boundaries, and complex data structures.
- Avoid unnecessary object copies in hot paths or large data transformations.
- Validate external input at the boundary.
- Prefer dataclasses, TypedDict, Pydantic models, or project-standard schema objects when structure matters.

## Tests

- Add or update tests for changed behavior, edge cases, and failure paths.
- Follow `rules/python-tests.md` for test execution and coverage expectations.

## Logging

- Log enough context to diagnose failures, but do not log secrets or raw sensitive data.
- Do not replace error handling with logging only. Either recover intentionally or propagate the error.
- Prefer structured or consistent log fields when existing project code uses them.

## Dependencies

- Do not add new dependencies without first checking whether the standard library or existing project dependencies already solve the problem.
- Prefer small local helpers over new dependencies for simple behavior.
- If adding a dependency is necessary, explain why existing options are insufficient.

## Secrets

- Do not expose API keys, access tokens, credentials, cookies, private keys, or secret values in logs, stdout, stderr, commits, shell history, examples, test fixtures, or generated reports.
- Do not run commands that expand secrets into shell history, such as `echo $TOKEN`.
- Do not enable `set -x` when commands may include secret-bearing values.

## Environment variables in tests

- When tests must temporarily modify `os.environ`, restore the previous environment in a `finally` block or use a test fixture such as `monkeypatch`.
- Do not mutate global environment state in a way that can leak across tests.
